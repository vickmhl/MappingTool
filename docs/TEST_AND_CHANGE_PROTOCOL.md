# Mapping 工具测试与改动协议

这份文档用于约束后续迭代，目标是减少“看一眼不满意，再小修一轮”的消耗。任何改动都必须先判断它属于哪一层，再用对应验收标准判断是否可以发布。

## 一、第一性原则

HRBP 使用这个工具不是为了看技术过程，而是为了把零散、脏乱、敏感的竞对信息，快速变成可校对、可检索、可汇报的组织结构。

因此产品必须坚持：

1. 界面只展示结果，不展示思考过程。
2. 上传页只解决两个动作：上传资料，打开虚拟演示 Demo。
3. 确认页只解决一件事：把不确定信息变成可入图信息。
4. 组织图分两类任务：汇报模式看结构，招聘模式找人。
5. 汇报模式默认只展示高层到下两级，不暴露海量人员。
6. 招聘模式允许展开、搜索、聚焦、折叠，用于一线 mapping。
7. 导出物必须可汇报，不能只是把混乱画布截图塞进 PPT。
8. 所有敏感资料必须保持本地处理，不依赖外部服务。

## 二、改动分层

每次需求先归类，不能直接动视觉或堆按钮。

### 1. 数据层

涉及上传、文本解析、PPTX 解析、OCR、候选生成、去重、证据来源、确认入库。

必须回答：

1. 输入是什么文件或文本。
2. 期望抽取哪些实体：人、部门、岗位、上级、团队规模、变化、备注。
3. 不确定信息如何进入人工确认，而不是静默丢失。
4. 重复人员如何提示，而不是直接覆盖。

验收标准：

1. 脏文本中至少能识别姓名、部门、岗位、上级中的三类信息。
2. PPTX 中可选文字必须优先解析。
3. 图片和 OCR 失败必须提示人工补录。
4. 候选项必须带证据片段。
5. 确认入图后组织图可立即反映变化。

### 2. 结构层

涉及组织图节点集合、层级、布局、展开折叠、搜索、筛选。

必须回答：

1. 当前是汇报模式还是招聘模式。
2. 当前是常规架构图还是树状图。
3. 默认应该展示多少节点。
4. 哪些节点被隐藏，用户如何展开。
5. 搜索和筛选是否真实改变结果。

验收标准：

1. 汇报模式默认只展示 L0、L1、L2，不展示 L3 及更深层级。
2. 汇报模式默认一级组织不超过 4 个，展开更多后可以增加。
3. 招聘模式可以展示更多人员，但必须按部门聚合，避免长横向铺开。
4. 树状图节点不能互相重叠。
5. 搜索结果不能被普通可见节点上限静默截断。
6. 管理者筛选必须只留下有下属或团队规模的人。
7. 画布不能出现页面级横向滚动条。

### 3. 视觉层

涉及颜色、卡片、线条、信息层级、按钮、右侧面板、画布密度。

必须回答：

1. 主信息是什么，辅助信息是什么。
2. 哪些颜色表示交互，哪些颜色表示状态。
3. 线条是否压住文字。
4. 卡片是否有足够留白。
5. 高亮态是否满足对比度。

验收标准：

1. 不允许白字白底、蓝字蓝底、弱对比浅灰字。
2. 连线不得穿过节点正文。
3. 节点卡片之间不得重叠。
4. 汇报模式节点只展示部门、一号位、人数、备注入口。
5. 招聘模式节点可以展示候选人、岗位、部门、团队信息，但必须可扫读。
6. 顶部工具栏不挤压画布，不产生横向滚动。
7. 右侧补图面板不遮挡主要操作，也不制造页面级横向滚动。

### 4. 导出层

涉及 PNG、PPTX、加密项目包。

必须回答：

1. 导出给谁看：老板、HRD、业务负责人、招聘同事。
2. 第一页要讲什么结论。
3. 组织图页是否能单独阅读。
4. 是否包含来源、备注、风险提示。

验收标准：

1. PPT 第一页必须是组织总览，不是画布截图。
2. 组织图页必须有标题、视图说明、节点统计。
3. 组织图截图只能作为结构页内容，不得占据整个报告逻辑。
4. PNG 导出应使用干净画布样式，不显示 React Flow 水印、控件、临时面板。
5. 导出前如果当前图过密，应提示切换汇报模式或降低节点上限。

## 三、固定测试矩阵

每次发布前必须完成以下测试。

### 自动测试

运行：

```bash
npm run test
npm run build
```

必须覆盖：

1. 文本抽取规则。
2. PPTX 文本解析。
3. 图片 OCR 失败后的人工补录路径。
4. 人员去重和冲突提示。
5. 汇报模式默认节点层级。
6. 招聘模式展开和搜索。
7. 筛选条件真实影响结果。
8. PPTX 导出第一页和组织图页。

### 本地浏览器验收

本地预览后检查：

1. 首屏只看到上传资料区和虚拟演示 Demo。
2. 不出现：首页、项目名称、下一步建议、工作流程、解析过程、资料体检清单。
3. 点击虚拟演示 Demo 后直接进入组织图。
4. 默认组织图不出现卡片重叠。
5. 默认组织图不出现线条穿字。
6. 切换汇报模式和招聘模式后，节点数量和内容明显不同。
7. 切换常规架构图和树状图后，布局明显不同。
8. 搜索一个已知姓名，画布只聚焦相关节点和必要上下级。
9. 点击人员详情后有明确关闭或返回入口。
10. 浏览器没有页面级横向滚动条。

### 公网页面验收

GitHub Pages 发布后检查：

1. 打开 `https://vickmhl.github.io/MappingTool/`。
2. 确认首页内容与本地一致。
3. 打开 Demo，确认组织图节点数在预期范围内。
4. 再次检查旧文案不存在。
5. 抽查导出 PPTX 可以打开。

## 四、缺陷处理方式

收到“很丑”“不能用”“很乱”这类反馈时，不直接猜改法，先转成可测试缺陷。

常见转译：

1. 很丑：检查对比度、字号层级、留白、按钮密度、卡片阴影、线条颜色。
2. 很乱：检查节点数量、层级深度、连线路由、默认缩放、是否有聚合。
3. 不能用：检查用户目标是否一屏内可完成，按钮是否可见，操作后是否有结果。
4. 导出不能汇报：检查 PPT 第一页是否有结论，图页是否可读，截图是否过密。
5. 筛选没用：检查筛选前后节点数量、人员集合、层级是否变化。

## 五、发布门禁

以下任一情况出现，不发布：

1. `npm run test` 失败。
2. `npm run build` 失败。
3. 首页出现已删除的旧文案。
4. 汇报模式默认出现 L3 或更深节点。
5. 默认画布有卡片重叠。
6. 默认画布产生页面级横向滚动。
7. PPT 第一页仍是画布截图。
8. GitHub Actions 未通过。

## 六、后续版本策略

后续不要按单个小按钮上线。每轮版本只处理一个主题：

1. 上传解析质量。
2. 确认入图效率。
3. 组织图结构和视觉。
4. 通话补图工作流。
5. 导出汇报质量。

每轮主题完成后，再统一发布。这样可以减少 token 消耗，也避免局部修改破坏整体体验。
